> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Corriger les bugs PWA (icône iOS cassée, manifest malformé) et réduire le bundle JS initial pour améliorer LCP/INP sur la home page.

**Architecture:** Deux changements indépendants — (1) PWA : créer `app/apple-icon.tsx` via la convention Next.js 14 pour générer l'icône iOS en PNG, corriger `manifest.json` et bumper le cache SW ; (2) CWV : supprimer les imports morts et lazy-loader les composants below-the-fold avec `dynamic()` pour réduire le JS bloquant.

**Tech Stack:** Next.js 14 · `next/og` (ImageResponse, déjà utilisé pour og:image) · `next/dynamic` · TypeScript

## Contexte codebase

- `app/layout.tsx` — référence `/icons/icon-192.svg` comme `apple-touch-icon` → **bug iOS** : Safari ignore les SVG pour les icônes home screen.
- `public/manifest.json` — icons avec `"sizes": "192x192"` sur des SVG (devrait être `"any"`) et `"purpose": "any maskable"` sur une seule entrée (devrait être deux entrées séparées).
- `public/sw.js` — cache nommé `keza-v1`, à bumper après changement d'assets.
- `app/page.tsx` — importe `PopularRoutes` (ligne 11) qui **n'est jamais utilisé** dans le JSX. Importe aussi `MultiDateCompare`, `PushNotifBanner`, `MilesCalculatorWidget`, `ProgramsWidget`, `HowItWorks`, `PromoBanner` — tous chargés eagerly alors qu'ils sont below-the-fold ou conditionnels (après search).
- `app/apple-icon.tsx` — **n'existe pas** → convention Next.js 14 : ce fichier génère automatiquement `/apple-touch-icon.png` et ajoute le `<link>` dans le `<head>`.
- `lib/og-templates.tsx` — `OG_WIDTH`, `OG_HEIGHT`, `ogWrapper()`, `ogTopBar()`, `ogBottomBar()` déjà créés (Sprint 4). L'apple-icon **ne doit pas** utiliser ces helpers (taille différente, style différent).

  return new ImageResponse(

      <div
        style={{
          width: 180,
          height: 180,
          display: "flex",
          alignItems: "center",
          justifyContent: "center",

          borderRadius: 38,

            display: "flex",

              lineHeight: 1,

    ),
    { width: 180, height: 180 }
  );
}

## Task 2 : Core Web Vitals — bundle reduction

**Files:**
- Modify: `app/page.tsx`

**Contexte :** `app/page.tsx` est `"use client"`. Il importe 17 composants en eager. `dynamic()` de Next.js fonctionne dans les client components — il reporte le chargement du bundle JS du composant à la première render, réduisant le JS initial parsé au chargement. Les composants below-the-fold ou conditionnels sont les meilleurs candidats.

**Composants à lazy-loader :**
- `MultiDateCompare` — s'affiche uniquement après une recherche (`hasSearched && !loading && results.length > 0`)
- `PushNotifBanner` — toujours en bas de page, non critique
- `HowItWorks` — section explicative, below-the-fold
- `PromoBanner` — below-the-fold
- `MilesCalculatorWidget` — widget below-the-fold (visible dans la sidebar)
- `ProgramsWidget` — widget below-the-fold (sous MilesCalculatorWidget)

**Composant à supprimer :**
- `PopularRoutes` — importé ligne 11 mais **jamais utilisé** dans le JSX.

- [ ] **Step 1 : Mettre à jour les imports dans `app/page.tsx`**

Remplacer le bloc d'imports des composants (lignes 1-27 actuelles) par la version avec `dynamic` :
